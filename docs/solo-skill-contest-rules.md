# TRAE SOLO 技能创作赛规则沉淀

> 来源：TRAE 官方社区规则帖  
> https://forum.trae.cn/t/topic/16860

## 1. 赛事定位

「一切皆可 Skill｜SOLO 技能创作赛」是 TRAE 官方主办的 Skill 创作与测评活动，鼓励用户使用 TRAE SOLO 把真实工作流、内容方法、效率工具、创意玩法沉淀为可执行、可分享、可复用的 Skill。

本次活动面向所有对 Skill 创作、AI 应用和真实场景实践感兴趣的用户开放，不要求必须具备技术背景。

## 2. 参赛赛道

### 2.1 Skill 创作赛道

适合人群：

- 产品
- 运营
- 数据分析师
- 设计师
- 程序员
- 自媒体从业者
- 其他有真实工作流或创意玩法的人

作品方向建议：

- 效率工具：日报自动生成器、会议纪要智能整理、邮件或消息批量处理、文件自动归类与命名、社群或私域运营助手、活动复盘助手等。
- 数据洞察：销售数据异常预警、竞品监控日报、用户反馈收集与分析、A/B 测试报告自动化等。
- 内容创作：多平台文案适配、短视频脚本生成、行业热点追踪、选题推荐、产品说明书自动生成等。
- 开发辅助：API 文档自动生成、代码 Review 助手、错误日志智能解析、技术方案模板生成器等。
- 开放创意：AI 角色扮演、互动游戏 Skill、个人知识库问答助手、面试官、教练、心理咨询等。

### 2.2 Skill 测评赛道

适合人群：

- 普通用户
- AI 爱好者
- 愿意认真体验和反馈 Skill 的用户

参与方式：

1. 在论坛选择一个感兴趣的参赛作品，不能是自己投稿的作品。
2. 实际安装、运行、体验该 Skill。
3. 发布一篇测评帖，分享使用过程、效果、问题和建议。
4. 优质测评会影响最终评奖和官方推荐。

测评帖要求：

- 至少包含一个参赛 Skill 的完整测评过程。
- 说明使用效果、亮点、问题和改进建议。
- 给出综合评价。
- 测评观点需要真实、中肯，严禁 AI 生成式空泛评价。
- 加分项：横向对比多个同类 Skill。

## 4. 有效参赛条件

必须满足：

- 作品基于 TRAE SOLO 开发。
- 必须在 TRAE 官方社区的「SOLO技能创作赛」专区发帖。
- 每个作品单独发帖。
- 作品需原创，不得搬运、抄袭或侵犯他人知识产权。
- 内容不得违反法律法规。
- 不得恶意灌水。
- 不得过度依赖 AI 生成无价值内容。
- 同一作者可以提交多个不同作品，每个作品可重复获奖。

推荐同时完成：

- 在发帖后提交官方飞书问卷。
- 如参与社媒传播奖，提交社媒链接。
- 保留真实使用截图、录屏、测试过程和迭代记录。

## 5. Skill 创作赛道作品要求

参赛帖必须包含：

- Skill 简介。
- 使用场景。
- 创作过程，可附提示词、工作流图。
- 效果展示。
- Skill 分享链接，且链接需要可正常访问和使用。
- 不少于 300 字原创内容。
- 真实使用截图或录屏。

建议补充：

- 目标用户是谁。
- 解决了什么真实痛点。
- 使用前后对比。
- 关键能力边界。
- 输入输出示例。
- 安装和使用步骤。
- 失败场景或注意事项。
- 后续优化计划。

## 6. 主奖项评审规则

主奖项经过两轮评审：

1. 初审：TRAE SOLO 对所有参赛作品进行初步筛选，并结合社区互动数据，筛选出 TOP 100 入围作品。
2. 终审：TRAE 产品、研发和运营专家组成官方评审团，对入围作品最终评审。

评分维度：

| 维度 | 权重 | 关注点 |
| --- | ---: | --- |
| 实用性 | 30% | 是否解决真实痛点；应用场景是否明确且有代表性 |
| 完成度 | 25% | 内容是否完整清晰；Skill 是否可稳定复现；用户引导是否友好 |
| 创新性 | 25% | 想法是否新颖有趣；是否带来新的视角或解决方案 |
| 社区影响力 | 20% | 是否获得真实讨论和正向反馈；投票和有效互动量综合表现 |

注意：

- 刷评论和刷投票会取消评奖资格。
- 社区互动是参考因素，但必须是真实互动。

## 9. 面向开发的作品策略

优先选择满足以下条件的 Skill 主题：

- 有清晰用户和真实场景。
- 使用前后差异明显，便于截图或录屏展示。
- 可以稳定复现，不依赖模糊口述。
- 输入输出边界清楚。
- 能体现 TRAE SOLO 的自动化或工作流价值。
- 有一定传播点，普通用户能快速理解。
- 不是泛泛的提示词合集，而是能完成任务的流程化 Skill。

开发时重点打磨：

- 入口提示是否清晰。
- 用户需要准备什么材料。
- Skill 每一步的产出是否可检查。
- 异常输入时是否有兜底说明。
- 生成结果是否有结构化输出。
- 是否能给出可直接使用的文件、文案、报告、表格或行动清单。
- 是否有真实测试样例。

## 10. 发帖模板

```markdown
# 【Skill 创作】作品名称

## 一句话介绍

这个 Skill 用来解决什么问题，适合谁使用。

## 适用场景

- 场景 1
- 场景 2
- 场景 3

## 痛点说明

不用这个 Skill 时，用户原来的流程是什么，有哪些低效、重复或容易出错的地方。

## Skill 能力

- 能力 1
- 能力 2
- 能力 3

## 使用方式

1. 准备什么输入。
2. 如何启动 Skill。
3. Skill 会如何处理。
4. 最终会输出什么。

## 使用效果展示

放真实截图或录屏链接，建议包含：

- 输入材料
- 运行过程
- 输出结果
- 使用前后对比

## 创作过程

说明从想法到成品的过程，可以包含：

- 为什么想到这个方向
- 关键提示词
- 工作流设计
- 遇到的问题
- 如何迭代优化

## Skill 分享链接

填写可正常访问和使用的 Skill 链接。

## 后续计划

- 优化方向 1
- 优化方向 2
- 优化方向 3
```

## 12. 当前开发结论

如果目标是冲击主奖项，作品不应只追求「能用」，而应同时满足：

- 有明确目标用户。
- 有可解释的真实问题。
- 有稳定可复现的 Skill 工作流。
- 有前后对比强烈的效果展示。
- 有完整、具体、原创的社区发帖内容。
- 有适合传播的案例或使用故事。

后续开发 Skill 时，可以先用本文件的「作品策略」和「提交前检查清单」筛选方向，再用「发帖模板」准备参赛材料。
